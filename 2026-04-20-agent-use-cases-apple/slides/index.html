§ Case 02 · 三个例子 006 / 015

都是一个 session 写完的

从"手痒"到 pip install。

gnvitop

多机 GPU 监控。
把 nvidia-smi 聚合成 JSON，
我的 agent 排队前会先读它。

bibtex-checker

投稿前校验 .bib，
对照 OpenAlex / Crossref 找死链和版本漂移。
一个典型 .bib 通常能捞出 20+ 处。

paper2poster

PDF → 打印就绪的会议 poster（HTML）。
省掉两天 InDesign 调版。

关键不是"这些工具多新"
而是决策规则反了：重复两次就值得做。

# 主题: agent RL > 排一条 10 步的学习路径。 > 每一步告诉我： > · 读哪一篇 > · 读完能做什么 > · 我应该能回答什么问题 > · 两句话的摘要 > > 按 因果顺序排，不按年份。

§ Case 04 · 回路 010 / 015

三个角色，一个下午

ideation → pilot → verdict。

  1. Framing把想法写成一句"能被实验推翻"的话。
  2. PilotLab agent 写代码，挑一张空闲 GPU，跑一遍最小规模。
  3. VerdictReviewer agent 给结论：KILL / PIVOT / CONTINUE。

最近一次 PIVOT 省掉了两周的 scaling。
Agent 是仪器，不是 collaborator —— 决策始终在我。

§ 共性 014 / 015

五个 case 共享的三件事

三个不变量。

01
Agent 是仪器，
不是同事。

taste、framing、取舍仍然在我。
它做的是我懒得写的那部分。

02
赢在尝试的
吞吐量。

一上午跑五次烂的，
好过一周只跑一次"很仔细的"。

03
好 prompt 描述
的是形状。

"给我 10 篇"很弱；
"按因果顺序 + 前置依赖"永远更好。
