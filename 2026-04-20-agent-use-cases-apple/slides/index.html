Case 1 — stay current

The daily AI briefing.

Every morning, an agent reads HuggingFace Trending, arXiv, and big-lab release feeds, clusters the noise, and hands me a single file I actually read with my coffee.

信息爆炸不是新问题，新的是治理方式： 把"订阅 → 过滤 → 摘要 → 归档"变成一条每天自动跑的 agent pipeline。

Case 01 · ai-daily stay current without drowning
§ Case 01 · mechanics 004 / 016

How it actually runs

A cron, a skill, and a knowledge base.

  • Trigger. Cron fires /ai-daily each morning.
  • Gather. Agent pulls HF Trending, arXiv categories, lab blogs.
  • Cluster. Group by theme, dedup near-duplicates, rank by novelty.
  • Write. One markdown briefing, ≤ 6 stories, each with a "why it matters".
  • Archive. Drop into ~/ai-knowledge/daily/, linked to a web dashboard.
# 2026-04-20 briefing $ claude /ai-daily → scanning 9 sources ... 312 items → clustering ............ 7 themes → ranking by novelty .... done ▸ theme 01 agent-RL post-training ▸ theme 02 long-context serving ▸ theme 03 on-device diffusion ... 4 more ✓ wrote ~/ai-knowledge/daily/2026-04-20.md ✓ updated dashboard (+7 cards)
Runtime ≈ 3 min No UI to maintain Same skill · different cron · team briefings

Case 2 — small tools

Utilities I used to
never bother to write.

The threshold for "should I build this?" has collapsed. Things that used to be "meh, I'll do it by hand" are now a 20-minute agent session and a pip-installable package.

工具的建造成本低到可以忽略之后， 你的行为会变 —— 每一次手动重复，都变成"值得写一个工具"。

Case 02 · small tools the threshold collapsed
§ Case 02 · examples 006 / 016

Two tools I ship and actually use

From paper cut to pip install.

gnvitop

A multi-host GPU monitor for my lab fleet. JSON output that agents can read before scheduling a job.

  • Aggregates nvidia-smi across 6 hosts
  • One command, one JSON blob
  • Consumed by my own agents before any training run

bibtex-checker

Catches dead DOIs, mismatched venues, arXiv-vs-published version drift before every paper submission.

  • Cross-checks against OpenAlex + Crossref
  • Flags 20–30 issues per typical .bib
  • Written in a single afternoon session

paper2poster

PDF → print-ready conference poster (HTML). A skill, not a product — invokable from any chat.

  • Reads the paper, picks a style, lays it out
  • Single standalone HTML → print PDF
  • Replaces 2 days of InDesign fiddling

Lesson. Each of these would have been "nice-to-have, maybe next year" pre-agent. Post-agent, the decision inverts: if I'll do it twice, I build a tool.

Case 3 — learn fast

Onboarding to a
field in one evening.

Take "agent RL" — I had read maybe three papers. Asking an agent for a reading list is a weak prompt. Asking it to build me a learning trajectory is a strong one.

真正有用的不是"给我 10 篇论文"， 而是"按因果顺序排一条学习路径，标出每篇的作用、前置知识、和我看完之后应该能回答什么问题"。

Case 03 · learn-fast reading list → trajectory
§ Case 03 · output 008 / 016

Prompt shape matters more than model choice

A trajectory, not a list.

The prompt, roughly

# topic: agent RL > Give me a 10-step learning > trajectory. For each step: > · the one paper to read > · what it lets me do next > · what I should be able > to answer after reading > · 2 sentence summary > Order by causal dependency, > not by date.

What comes back

  • Step 01. RLHF fundamentals — InstructGPT, 2022
  • Step 02. Preference modeling without RL — DPO
  • Step 03. Rewarding reasoning, not answers — process rewards
  • Step 04. Agent-flavored RL — tool use as actions
  • Step 05. Self-play with verifiable rewards
  • ... five more, each with a "you'll be ready to ask X"

It's wrong in places. That's fine — it gives me a scaffold to argue with, which is faster than writing one from scratch. I've done this for three fields this year.

Case 4 — auto-research

Minimum idea,
minimum experiment.

Not "write me a NeurIPS paper." The useful shape is: take one small, clean claim — run the smallest experiment that could falsify it — report honestly what happened.

auto-research 的价值不在"自动写论文"， 而在缩短"从直觉到第一条信号"的距离：通常一个下午，一张图，一个决定 —— 继续还是放弃。

Case 04 · auto-research the minimum honest experiment
§ Case 04 · the loop 010 / 016

Five phases, one afternoon

Ideation → pilot → verdict.

  • Frame. Write the claim as one sentence an experiment can break.
  • Scope. Agent proposes the smallest dataset + model that could show signal.
  • Run. A Lab sub-agent writes code, dispatches to a free GPU, collects results.
  • Review. A separate Reviewer agent returns a verdict: KILL / PIVOT / CONTINUE.
  • Decide. I look at one figure and one paragraph, then I decide — not the agent.

Recent example

Claim: "Calibration-aware sampling improves multi-step agent traces on XYZ."

  • Scope: 2 tasks, 1 model, 3 seeds
  • Runtime: ~4 hours on one A100
  • Verdict: PIVOT — signal exists, but only on one task family
  • Cost saved: 2 weeks of scaling before knowing

The agent is an instrument, not a collaborator. It runs the experiment I was too lazy to script. I still own the idea and the interpretation.

Case 5 — vibe coding

Shipping a minimum
MCP in one afternoon.

The fastest way to know if a product idea has legs is to make the thinnest possible version of it usable by one real user — me. MCP is the honest interface for that: a single verb, a single data shape.

"Vibe coding" 不是草率 —— 是把验证周期从两周压到两小时。 你不需要创业，你需要那个"我其实不想用它"的诚实时刻尽早出现。

Case 05 · vibe-coding shortest honest validation
§ Case 05 · what emerged 012 / 016

From "I wish this existed" to a running MCP server

One verb, one week-one user.

The afternoon

  • 2:00 — sketched the one primitive the server must do
  • 2:30 — agent scaffolded the MCP tool definitions
  • 3:30 — auth, rate-limits, a tiny web dashboard
  • 5:00 — used it inside Claude Code for a real task
  • 6:00 — wrote a note: "which part do I actually keep opening?"

The lesson, inverted

最早的 prototype 最有价值的时刻，不是"它能跑"， 而是第二天我发现我没再打开它。 那才是最诚实的信号。

The agent makes building nearly free. That forces the interesting question back to where it belongs: is this worth using at all?

1 afternoon ~400 lines 1 honest "no" saved 3 months

§ Synthesis 014 / 016

What all five cases share

Three invariants.

  • 1. The agent is an instrument, not a colleague. I keep taste, framing, and the call on what matters. It does the work I was too lazy to script.
  • 2. The win is throughput of attempts, not quality of any one attempt. Five bad tries in a morning beat one careful try in a week — if each try is cheap to discard.
  • 3. Good prompts describe the shape, not the answer. "Give me a learning trajectory with preconditions" beats "give me 10 papers." Every time.

Implications · 对研究的影响

  • Small, ugly prototypes become the new literature review.
  • "I should build a tool" crosses the bar much more often — and that changes which problems you can afford to notice.
  • The real skill becomes deciding: what to scale, what to KILL, what was never worth starting.

Takeaways for the team

What I'd like to try with you.

Low-friction places to start

  • A shared daily briefing skill, tuned to the team's reading list
  • A lab-internal "paper cut" utility shelf — each utility is one session
  • A reviewer-agent pass in every pilot experiment, before we scale

What I'd love your pushback on

  • Where the "instrument, not colleague" framing breaks down
  • When an agent-built prototype actually slows a project
  • Guardrails for reproducibility when code is half-agent-written

我不是来推销 agent 的。我是来分享一套已经改变了我日常工作节奏的具体做法 —— 然后听听各位在 Apple Research 的语境下，会怎么用、怎么不用。
